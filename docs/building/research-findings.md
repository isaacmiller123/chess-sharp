## Stockfish integration in Electron (native NNUE binary vs stockfish.wasm; UCI control, MultiPV, strength sliders, info parsing, licensing, recommended integration)

**Summary.** For an offline Electron chess-analysis app, the clearly recommended path is to BUNDLE the native Stockfish 18 NNUE binary (Windows-first, with macOS/Linux builds for cross-platform) and SPAWN it from the Electron MAIN process via Node's child_process (or utilityProcess), talking UCI over stdin/stdout. This is faster than WASM (full native NNUE + true multithreading), and it completely SIDESTEPS the SharedArrayBuffer + COOP/COEP cross-origin-isolation requirement that multithreaded stockfish.wasm imposes on the renderer. Stockfish 18 (released 2026-01-31) is the current stable release; it is GPLv3 and ships with the NNUE network EMBEDDED in the official compiled binary, so NO separate .nnue file is needed at runtime (EvalFile defaults to the embedded net). The renderer (chessboard UI) communicates with the main process via IPC; the main process owns the engine child process. Use MultiPV for top-N lines, UCI_LimitStrength+UCI_Elo (1320-3190) and Skill Level (0-20) for adjustable-strength sliders, Threads/Hash for performance, and parse streaming 'info' lines (depth/seldepth/multipv/score cp|mate/nodes/nps/pv) until 'bestmove'. node-uci (MIT) is a reasonable helper for the analysis path, but a thin hand-rolled UCI wrapper is often better for live MultiPV streaming and stop()-on-the-fly. GPLv3 compliance for redistribution: you MUST ship the license text and provide the exact corresponding Stockfish source (or a written offer/pointer to it) alongside the binary.


**Key facts**
- Latest stable: Stockfish 18, released 2026-01-31 (SFNNv10 net). Prior: 17.1 (2025-03-30), 17 (2024-09-06). The 'stockfish' npm package and the download page already ship SF18 builds.
- License: Stockfish is GPLv3. Redistribution obligation: whenever you distribute it you MUST include the full GPLv3 license text and the exact corresponding source code (or a pointer/written offer to where it can be found) to build the binary you ship. Any modifications must also be GPLv3. The NNUE net is trained on Leela Chess Zero data under the Open Database License (ODbL); the bundled binary embeds it.
- NNUE network is EMBEDDED in the official compiled binary -> no separate .nnue file required at runtime. EvalFile defaults to the embedded net (nn-[hash].nnue). You can optionally point EvalFile at an external net, but for this app you do NOT need to ship a loose .nnue file.
- Windows binary variants (pick by CPU, in order of speed): x86-64-vnni512 / vnni256 (newest Intel w/ AVX-512 VNNI), bmi2 (Intel Haswell+/AMD Ryzen 5000+), avx2 (Haswell+/Ryzen 1000-3000), sse41-popcnt (older). Simplest robust choice: ship the x86-64-universal build, which auto-detects CPU capabilities at startup and runs the fastest available code path -- avoids shipping/selecting multiple variants.
- Native binary is spawned from Electron's MAIN process (or a utilityProcess) with child_process.spawn(path, [], { stdio: ['pipe','pipe','pipe'] }). Write UCI commands to stdin; read engine output by line-buffering stdout. The renderer must NOT spawn the process directly -- route commands/results over IPC (ipcMain/ipcRenderer or MessagePort).
- SharedArrayBuffer / cross-origin isolation: multithreaded stockfish.wasm REQUIRES the page to be cross-origin isolated (self.crossOriginIsolated === true), which needs Cross-Origin-Opener-Policy: same-origin and Cross-Origin-Embedder-Policy: require-corp headers (set via session.defaultSession.webRequest.onHeadersReceived or a custom protocol in Electron). The NATIVE-BINARY path AVOIDS this entirely -- no COOP/COEP, no SAB, no header plumbing. This is a major reason to prefer the native binary for a desktop app.
- UCI options + ranges: Hash spin default 16 (1..33554432 MB); Threads spin default 1 (1..1024); MultiPV spin default 1 (1..500); Skill Level spin default 20 (0..20); UCI_LimitStrength check default false; UCI_Elo spin default 1320 (range 1320..3190, anchored to CCRL 40/4 at 120s+1s); UCI_ShowWDL check false; Ponder check false; Move Overhead spin default 10 (0..5000 ms); EvalFile string (defaults to embedded net name).
- Adjustable strength: two mechanisms. (a) Realistic rating ladder -> setoption name UCI_LimitStrength value true; setoption name UCI_Elo value <1320..3190>. (b) Coarse 'how dumb' slider -> setoption name Skill Level value <0..20> (lower = weaker, occasionally plays a deliberately worse move). For a teaching app, UCI_Elo gives intuitive rating-matched opponents; Skill Level is good for beginner bots.
- MultiPV for top-N lines: setoption name MultiPV value N before 'go'. Each search then emits N 'info ... multipv 1..N ... pv ...' lines per depth; the multipv index tells you which ranked line it is (multipv 1 = best). Increasing MultiPV slows search, so use a small N (3-5) for the analysis panel and reset to 1 for pure best-move/play.
- 'info' line parsing: tokens of interest -> depth <d>, seldepth <d>, multipv <i>, score cp <centipawns> | score mate <plies>, nodes <n>, nps <n>, time <ms>, hashfull <permil>, tbhits <n>, pv <move1 move2 ...>. score is from side-to-move POV (negate for absolute White-centric eval). 'score cp' is centipawns (100 = 1 pawn); 'score mate N' = mate in N moves (sign = who mates). Search ends with 'bestmove <move> [ponder <move>]'.
- 'go' termination modes: go depth <d> (fixed depth, deterministic, good for consistent analysis), go nodes <n> (fixed node budget, hardware-independent reproducibility), go movetime <ms> (fixed wall-clock, good for responsive UI / time-budgeted hints), go infinite (search until 'stop'; use for live interactive analysis where the user moves pieces). Tradeoff: depth gives quality-consistent but variable-time results; movetime gives time-consistent but quality-variable; nodes gives the most reproducible cross-machine results.
- Helper packages: node-uci (MIT, ebemunk) wraps a UCI engine with Engine(path)/init()/setoption()/position()/go({depth|nodes|movetime}) returning a parsed result incl. info+bestmove -- good for one-shot analysis. The 'stockfish' npm package and lichess-org/stockfish.wasm / stockfish.js are the WASM/JS route (single-threaded variants like stockfish-18-single.js run WITHOUT COOP/COEP; multithreaded needs SAB).

**Recommendations**
- Architecture: bundle the native Stockfish 18 binary and spawn it from the Electron MAIN process (use utilityProcess or child_process.spawn). Keep one persistent engine process; expose analyze/play/stop over IPC to the renderer. Do NOT use multithreaded WASM in the renderer for a desktop app -- it only adds SharedArrayBuffer + COOP/COEP complexity for worse performance.
- Binary packaging: ship the x86-64-universal Windows build (auto-detects CPU at startup) to avoid maintaining multiple per-CPU variants; place it under an unpacked path (electron-builder asarUnpack / extraResources) because asar-packed binaries are not executable. Resolve its path at runtime from process.resourcesPath. Ship matching native builds for macOS (Apple Silicon arm64 + Intel) and Linux. Mark the file executable (chmod +x) on non-Windows during build/postinstall.
- Engine session protocol: on spawn send 'uci' -> wait for 'uciok'; send setoption lines (Threads = max(1, physicalCores-1), Hash = e.g. 128-512 MB depending on RAM); send 'isready' -> wait 'readyok'; send 'ucinewgame' at game/position reset. Per analysis: 'position fen <FEN>' (or 'position startpos moves ...'), then 'go ...'. Always send 'stop' before issuing a new 'go' on a still-running infinite search.
- Line handling: buffer stdout and split on newlines (UCI is line-oriented; a single chunk may contain partial/multiple lines). Tokenize each 'info' line by spaces; treat everything after the 'pv' token as the move list. Track the latest line per multipv index so the analysis panel shows N stable rows that update by depth.
- Strength sliders for the teaching app: expose a rating slider mapped to UCI_LimitStrength=true + UCI_Elo (1320-3190); optionally a separate 'beginner bots' set using Skill Level 0-20. Remember to set UCI_LimitStrength=false (and Skill Level 20) when the user wants full-strength analysis/coaching.
- Analysis vs play modes: for the live analysis board use MultiPV 3-5 + 'go infinite' (or 'go movetime') and stream lines; for engine-vs-human play use MultiPV 1 + a bounded 'go movetime'/'go depth' for snappy moves. Reset MultiPV to 1 outside the analysis panel to keep search fast.
- Helper choice: use node-uci (MIT) for simple request/response analysis to move fast, OR write a ~150-line UCI wrapper for full control over live MultiPV streaming, mid-search 'stop', and ponder. Given this app's live-analysis + coaching needs, a thin custom wrapper around child_process is recommended; keep node-uci as a fallback/reference.
- GPLv3 compliance (ship-blocking): include the verbatim GPLv3 LICENSE file with the app, display an attribution/credits screen, and provide the exact corresponding Stockfish 18 source (bundle a source tarball in resources, or include a written offer + pinned commit/release URL). Because the bundled net is ODbL (Leela data), also credit Leela Chess Zero. Since you ship the unmodified official binary, the cleanest compliance is: pin the exact official release tag and link/bundle that source.

**Libraries**
- **Stockfish 18 (native binary)** (GPLv3 (engine); bundled NNUE net derived from Leela data under ODbL. Redistribution requires shipping license + corresponding source.) — Core analysis/coaching engine. Bundle the native NNUE binary (x86-64-universal recommended for Windows; per-CPU variants vnni512/bmi2/avx2/sse41-popcnt available). NNUE net embedded -> no separate file at runtime. Not recommended as primary for a desktop app. https://github.com/lichess-org/stockfish.wasm

**Data sources**
- **Stockfish official release binaries (Windows/macOS/Linux, per-CPU variants + x86-64-universal)** (GPLv3, Compiled executables (.exe on Windows) with NNUE net embedded) — https://github.com/official-stockfish/Stockfish/releases Single binary per platform/variant, on the order of tens of MB; one file, no separate net needed at runtime
- **Stockfish evaluation networks (NNUE) repo** (Net trained on Leela Chess Zero data under ODbL, .nnue network files (only needed if NOT using the embedded net)) — https://github.com/official-stockfish/networks Big net ~tens of MB; not required to ship separately since official binaries embed the matching net

**Risks**
- GPLv3 is the main distribution risk: shipping Stockfish (even unmodified, even in a paid app) obligates you to provide the corresponding source and license. If the rest of your app is proprietary, keep Stockfish as a separately-distributed binary invoked via UCI (arms-length 'aggregation'/separate process) rather than statically linking engine code into your app, and document it -- failing to include source/license is the most common real-world violation.
- Native binary packaging pitfalls in Electron: binaries inside the asar archive cannot be executed; you must unpack (asarUnpack/extraResources) and the runtime path differs between dev and packaged builds (process.resourcesPath). On macOS the binary must be code-signed/notarized and may be killed by Gatekeeper; on macOS/Linux it must have the execute bit set. These cause 'works in dev, fails when packaged' bugs.
- CPU-variant mismatch: shipping only a bmi2/avx2/vnni build will crash with an illegal-instruction error on older CPUs. The x86-64-universal build mitigates this; if you ship specific variants you must detect CPU features and pick a safe fallback (sse41-popcnt).
- stockfish.wasm multithreaded path (if used as fallback for unsupported platforms) requires cross-origin isolation: you must inject COOP: same-origin + COEP: require-corp headers in Electron (session.webRequest.onHeadersReceived or a custom file:// protocol handler), and verify self.crossOriginIsolated===true. Misconfiguration silently disables SharedArrayBuffer and forces single-thread or breaks the engine. Avoid by using the native binary.
- Process lifecycle leaks: the engine child process must be killed on window close/app quit, and a runaway 'go infinite' must be 'stop'ped before reuse, or you leak CPU and zombie processes. Buffer-splitting bugs (assuming one stdout chunk == one line) cause dropped/garbled info lines.
- Version drift: Stockfish 18 is current (2026-01-31) but updates frequently; node-uci is mature but lightly maintained, and 'score'/'wdl' semantics or new UCI options can change between releases -- pin the engine version you ship and the corresponding source for compliance and reproducibility.
- Hash/Threads misconfiguration: defaults (Hash 16MB, Threads 1) leave huge performance on the table for a desktop analysis app; but setting Hash too high or Threads above physical cores can degrade performance or exhaust RAM. Size them from the user's machine at runtime.

## Open chess visual/audio assets (piece sets, board textures, sounds)

**Summary.** The lichess-org/lila repo bundles 42 piece sets (public/piece/*), board textures (public/images/board/*), and ~10 sound themes (public/sound/*), all documented with per-asset licenses in COPYING.md (https://raw.githubusercontent.com/lichess-org/lila/master/COPYING.md). Each piece set is exactly 12 SVG files named {w,b}{P,N,B,R,Q,K}.svg (e.g. wP.svg, bN.svg, wK.svg). The board UI is rendered by chessground (https://github.com/lichess-org/chessground), which is GPL-3.0 — a critical copyleft constraint for the whole app. Themes are pure CSS: a wrapper class selects piece images via selectors like `.merida cg-board piece.pawn.white { background-image: url('.../merida/wP.svg') }`, and board backgrounds via `.blue cg-board { background-image: url('.../board/blue.svg') }`. CRITICAL LICENSE SPLIT: roughly half the piece sets (all of sadsnake1's: staunty, maestro, fresca, cardinal, icpieces, gioco, tatiana, dubrovny; plus horsey, california, caliente, anarcandy, disguised, cooke, monarchy, xkcd) are CC BY-NC-SA 4.0 — NON-COMMERCIAL, must be excluded from a commercial/sold app. The truly redistribution-safe pieces are: cburnett & mono (GPLv2+), merida (GPLv2+), chessnut (Apache 2.0), fantasy/spatial/celtic (MIT, Maurizio Monge), rhosgfx (CC0), shapes (CC BY-SA 4.0), mpchess (GPLv3+), and kiwen-suwi/Firi/totoy/papercut (CC BY 4.0). For SOUNDS, the default 'standard' move/capture/check set is NOT in the free-exceptions list — it falls under COPYING.md's non-free 'The other sounds in public/sound' clause (no clear license, Lichess staff never clarified redistribution). Only futuristic, nes, piano, sfx are cleanly free (AGPLv3+, by Enigmahack); lisp is CC BY-NC-SA (non-commercial). For a commercial offline app, do NOT ship Lichess 'standard' sounds — use Kenney CC0 audio (kenney.nl) or generate/record your own. NOTE: AGPLv3 board/sound assets and GPL pieces impose copyleft on bundling code; CC-licensed assets (CC0/CC BY/CC BY-SA/MIT/Apache) are cleaner for an Electron app that you don't want forced under (A)GPL.

## The Lichess open puzzle database (database.lichess.org puzzles section)

**Summary.** Lichess publishes its entire puzzle collection as a single CC0-licensed (public-domain) CSV compressed with Zstandard at https://database.lichess.org/lichess_db_puzzle.csv.zst. Verified live via HTTP HEAD on 2026-06-24: the file is 299,950,785 bytes (~286 MiB / ~300 MB), Last-Modified 2026-06-03, containing 6,014,381 puzzles (the official page states "6,014,381 chess puzzles, rated, and tagged"; last data refresh 2026-06-04). Decompressed it is roughly 2 GB of plain CSV (~7.1x ratio). The exact 10-column header is: PuzzleId,FEN,Moves,Rating,RatingDeviation,Popularity,NbPlays,Themes,GameUrl,OpeningTags. CRITICAL ENGINEERING NOTE for ingestion: Moves are in UCI (long algebraic, e.g. e2e4, e7e8q for promotion). The FEN is the position BEFORE the opponent's setup move; the FIRST move in Moves is the opponent's move that creates the puzzle (apply it to the FEN to get the position you show the player), and the SECOND move onward is the actual solution the player must find. CC0 means it is fully redistributable and can be bundled offline in the Electron app with no attribution requirement (attribution is still courteous). This is the ideal, fully-open puzzle source for an offline app — no proprietary equivalent needed. The full machine-readable theme taxonomy (Themes column is a space-separated list of theme keys) and a complete SQLite ingestion plan are documented below. A known gotcha: Lichess compresses with a large zstd window, so decompression requires the long-distance-matching flag (zstd --long=31 -d), or you will hit "Frame requires too much memory for decoding".


**Key facts**
- DOWNLOAD URL: https://database.lichess.org/lichess_db_puzzle.csv.zst — single file, no auth, direct download. Index page: https://database.lichess.org/#puzzles
- LICENSE: Creative Commons CC0 1.0 Universal (public domain dedication). Freely redistributable; can be bundled inside the offline Electron app. No attribution legally required (courtesy attribution recommended).

- ROW COUNT: 6,014,381 puzzles (official page text: '6,014,381 chess puzzles, rated, and tagged'). Data last updated 2026-06-04. Grows over time — was ~4.6M in late 2024, now ~6.0M.
- DECOMPRESSED SIZE: ~2.0 GB plain CSV (Lichess states uncompressed is ~7.1x the compressed size; 300 MB * 7.1 ~= 2.1 GB).
- COMPRESSION: Zstandard (.zst). Streaming/partial-decompressable — you can cancel a download and still decompress the partial file. NOTE: compressed with a large window, so use 'zstd --long=31 -d' to avoid the 'Frame requires too much memory for decoding' error.

- MOVES ENCODING: UCI long-algebraic, space-separated (e.g. 'd3d6 f8d8 d6d8 f6d8'). Promotion appends piece letter (e.g. e7e8q). Use a chess library (chess.js / chessops / python-chess) to convert to SAN for display.
- FEN/SOLUTION SEMANTICS (most important for correct puzzle replay): The FEN is the position BEFORE the opponent's move. Apply Moves[0] (the opponent's setup/lead-in move) to the FEN to get the position shown to the player. Moves[1..] is the solution the player must play. The side to move in the *presented* position is therefore opposite to the side-to-move in the raw FEN.
- Rating = Glicko2 rating of the puzzle (each solve attempt treated as a rated game player-vs-puzzle). RatingDeviation = Glicko2 RD. Popularity = integer in [-100,100], 100*(up-down)/(up+down). NbPlays = number of times played.
- Themes column = SPACE-separated list of theme keys (e.g. 'crushing hangingPiece long middlegame'). OpeningTags = space-separated opening names, ONLY populated for puzzles that occur before move ~20 (else empty). GameUrl links to the source lichess game with a #ply anchor.
- PUZZLE GENERATION PROVENANCE: derived from analysis of ~600M Lichess games, positions re-analyzed with Stockfish NNUE at 40 meganodes; ratings via Glicko2.

**Recommendations**
- INGESTION PIPELINE (build-time, NOT runtime): (1) Download lichess_db_puzzle.csv.zst (~300 MB). (2) Decompress with 'zstd --long=31 -d lichess_db_puzzle.csv.zst' -> ~2 GB CSV. (3) Bulk-load into SQLite. (4) Build indexes. (5) Optionally PRUNE to a subset (e.g. NbPlays>=50 AND Popularity>=80) to keep the bundled DB small, then VACUUM. Ship the resulting .sqlite (or .sqlite.zst) inside the Electron app's resources.
- FAST SQLITE LOAD via CLI (.import is far faster than row-by-row inserts): create table, then use the sqlite3 dot-commands. Recommended schema:
CREATE TABLE puzzles (
  PuzzleId TEXT PRIMARY KEY,
  FEN TEXT NOT NULL,
  Moves TEXT NOT NULL,
  Rating INTEGER NOT NULL,
  RatingDeviation INTEGER,
  Popularity INTEGER,
  NbPlays INTEGER,
  Themes TEXT,
  GameUrl TEXT,
  OpeningTags TEXT
);
Then run sqlite3 with:
.mode csv
.import --skip 1 lichess_db_puzzle.csv puzzles
(the --skip 1 drops the header row). Wrap in 'PRAGMA journal_mode=OFF; PRAGMA synchronous=OFF;' for a ~2x faster one-shot import.
- INDEXES for theme + rating filtering: CREATE INDEX idx_rating ON puzzles(Rating); — for rating-band selection. Because Themes is a space-separated string, plain B-tree indexes can't match a single theme efficiently. Two robust options: (A) Build an FTS5 virtual table over Themes and query 'WHERE themes MATCH 'fork'' joined back by PuzzleId; or (B) normalize into a junction table puzzle_themes(PuzzleId TEXT, Theme TEXT) with INDEX(Theme, PuzzleId) and query 'JOIN puzzle_themes USING(PuzzleId) WHERE Theme=? AND Rating BETWEEN ? AND ?'. Option B gives the cleanest, fastest theme+rating filtering and is recommended for a teaching app that selects by single theme.
- THEME NORMALIZATION SCRIPT (one-time, build step): after import, run an INSERT...SELECT that splits the Themes string on spaces into puzzle_themes rows. In SQLite this is easiest done in a host language (Node better-sqlite3 / Python) iterating rows and inserting tokens; with ~6M rows averaging ~4 themes you get ~24M junction rows — index (Theme, Rating) for instant band queries. Consider storing Rating in the junction table too to make it a covering index: puzzle_themes(Theme, Rating, PuzzleId) so theme+rating queries never touch the main table.
- RUNTIME QUERY PATTERN for the coach (offline, deterministic, no LLM): SELECT a random puzzle for a target theme and player rating band, e.g. 'SELECT p.* FROM puzzle_themes t JOIN puzzles p USING(PuzzleId) WHERE t.Theme=:theme AND t.Rating BETWEEN :lo AND :hi ORDER BY RANDOM() LIMIT 1'. For reproducible/curriculum sequencing, order by a stable key (NbPlays DESC or PuzzleId) instead of RANDOM() and page through.
- ELECTRON PACKAGING: use better-sqlite3 (synchronous, fast, native) in the main process, NOT in the renderer. Ship the prebuilt DB read-only under app resources (extraResources) and open with 'readonly: true, fileMustExist: true'. Use chessops or chess.js to (a) apply Moves[0] to FEN for the presented position and (b) convert UCI -> SAN for the move list / annotations. All of this is fully offline.
- SIZE MANAGEMENT: the full 6M-puzzle DB is large (~1.5-2.5 GB depending on indexes). For a shipped app, filter aggressively at build time (e.g. keep ~300k-1M high-quality puzzles spanning all themes and rating bands), VACUUM, and optionally zstd-compress the .sqlite for the installer and decompress on first run. This keeps the installer lean while preserving theme/rating coverage.
- VALIDATION CHECK at build time: assert the header line matches exactly 'PuzzleId,FEN,Moves,Rating,RatingDeviation,Popularity,NbPlays,Themes,GameUrl,OpeningTags' before import, so a future schema change by Lichess fails loudly instead of silently corrupting column mapping.

## Opening data & explorer (offline)

**Summary.** An offline opening explorer for the app should be built from two redistributable pillars: (1) the lichess-org/chess-openings dataset for OPENING NAMES (CC0/public domain, ~3,733 named lines across volumes A-E), and (2) a Polyglot .bin opening BOOK for book-move suggestions and transposition-aware lookup (open, GPL-derived books exist; the format is trivial to read). Both are keyed on the same fundamental idea: a normalized position string. Lichess's own approach (scalachess OpeningDb) keys openings by an "opening FEN" = the 4-field FEN (piece placement, side to move, castling, en-passant) with NO move counters — identical to the EPD column the chess-openings build generates. Name detection is a longest/last-prefix match: walk every position of the game, look up each position's opening-FEN in a Map, keep the deepest match (lichess stops once the board drops below 20 pieces). Polyglot books key on a 64-bit Zobrist hash of the same position fields, giving you transpositions for free (different move orders reaching the same position collide on the same key). CRITICAL correctness rule, verified empirically with python-chess 1.11.2: the en-passant field in both EPD and the Zobrist hash is set ONLY when a legal en-passant capture actually exists in that position (after 1.e4 e5 2.Nf3 the EPD ep field is "-", not "e6"); a naive FEN that always emits the ep square will silently fail to match keys. For redistributable master/strong-player games, the Lichess open database (database.lichess.org) is the standout: explicitly CC0, "use them for... commercial purpose, anything you like." Most other "master" databases are either non-commercial (Lumbra's Gigabase, CC BY-NC-SA) or have unstated/ambiguous licenses (PGN Mentor, nikonoel Lichess Elite) and should be flagged.


**Key facts**
- chess-openings dataset (github.com/lichess-org/chess-openings): SOURCE files a.tsv..e.tsv have only 3 columns: eco, name, pgn (tab-separated, title-case names, PGN in SAN). License is CC0 1.0 (COPYING.txt is the full CC0 deed): 'As a collection of facts, this data set is in the public domain... released under the CC0 Public Domain Dedication.' Fully redistributable, commercial-OK, no attribution required.
- The uci and epd columns DO NOT exist in the source TSVs — they are generated by `bin/gen.py` (needs `pip install chess>=1,<2`) into a dist/ folder via `make`. dist output header is exactly: `eco\tname\tpgn\tuci\tepd`. uci = space-joined UCI moves (e.g. `e2e4 c7c5`); epd = `board.epd()` = 4-field FEN (placement, side-to-move, castling, en-passant). Verified by building locally: 3,733 named opening entries total (a=805, b=764, c=1244, d=566, e=354 data rows).
- EPD format is canonical position key. Confirmed example rows: Sicilian Defense (B20) epd `rnbqkbnr/pp1ppppp/8/2p5/4P3/8/PPPP1PPP/RNBQKBNR w KQkq -`; Ruy Lopez (C60) epd `r1bqkbnr/pppp1ppp/2n5/1B2p3/4P3/5N2/PPPP1PPP/RNBQK2R b KQkq -`. Note NO halfmove/fullmove counters and NO trailing 0 1. gen.py enforces EPD uniqueness (`by_epd` dict errors on duplicate), proving EPD is a safe primary key for name lookup.
- EN-PASSANT RULE (the #1 correctness gotcha, verified with python-chess 1.11.2): ep field is set ONLY if a legal ep capture exists. After 1.e4 e5 2.Nf3 -> EPD ep = '-' (has_legal_en_passant False). After 1.e4 Nf6 2.e5 d5 -> EPD ep = 'd6' (has_legal_en_passant True). Your offline key generator MUST replicate 'ep only if legal' or lookups miss. Polyglot Zobrist hashing applies the SAME rule (only XOR an ep-file random when an enemy pawn can actually capture).
- ECO system: 5 volumes A-E, each 00-99, codes like A00, B20, C60 (regex `^[A-E]\d\d$`). A=flank/irregular, B=semi-open (incl Sicilian/Caro), C=open (1.e4 e5) and French, D=closed/semi-closed Queen's Gambit, E=Indian defenses (Nimzo/King's Indian). ECO is a de-facto public classification standard (no single owning license); the chess-openings dataset's mapping of code->name is the CC0 redistributable artifact to ship.
- Polyglot .bin format: a flat array of 16-byte entries sorted ascending by key, looked up via binary search. Each entry, big-endian, struct '>QHHI': key=uint64 (8 bytes, Zobrist hash), move=uint16 (2 bytes), weight=uint16 (2 bytes, ~play probability), learn=uint32 (4 bytes, usually 0). Multiple consecutive entries can share a key (one per candidate move in that position).
- Polyglot 16-bit move decoding (from python-chess polyglot.py, authoritative): to_square = raw & 0x3f; from_square = (raw >> 6) & 0x3f; promotion = (raw >> 12) & 0x7 (0=none, else 1=N..4=Q, +1 offset). Squares are 0=a1..63=h8 (file = sq&7, rank = sq>>3). CASTLING QUIRK: encoded as king-captures-own-rook (e1->h1, e1->a1, e8->h8, e8->a8), NOT e1g1; you must rewrite these to standard king-two-square castling (e1g1/e1c1) before applying to a normal board.
- Polyglot Zobrist hash uses a fixed 781-entry random array (POLYGLOT_RANDOM_ARRAY): indices 0-767 = piece*64+square where piece_index=(piece_type-1)*2+color (so the array constants are part of the spec and identical across all compliant tools); 768-771 = castling (WK,WQ,BK,BQ); 772-779 = en-passant file (only if legal capture exists); 780 = turn (XORed when WHITE to move). hash = pieces ^ castling ^ ep ^ turn. python-chess's `chess.polyglot.zobrist_hash(board)` implements this exactly and is the easiest reference/oracle.
- Free/open Polyglot books: github.com/gmcheems-org/free-opening-books aggregates .bin books incl. Cerebellum Light (Brainfish), GM2001.bin, Komodo.bin, Rodent.bin (from donna_opening_books), Titans.bin & Human.bin (Saros). These are engine community books — generally freely downloadable but individual licenses vary and most are NOT formally CC0/MIT; treat as 'free to redistribute non-commercially / unclear' and prefer to GENERATE your own .bin from CC0 Lichess PGNs to guarantee clean licensing.
- Redistributable master-game PGN: Lichess open database (database.lichess.org) is explicitly CC0 — 'Database exports are released under the Creative Commons CC0 license. Use them for research, commercial purpose, publication, anything you like.' Includes monthly standard rated PGN dumps (.zst), 6,014,381-puzzle CSV, and 388M Stockfish position evaluations (JSONL). This is THE clean source — filter to high Elo to make a 'master-like' book/explorer with zero license risk.
- PROPRIETARY / NON-REDISTRIBUTABLE to FLAG: Chess.com opening explorer & game data — proprietary, not redistributable (open equivalent: Lichess CC0 DB + chess-openings names). Lumbra's Gigabase (lumbrasgigabase.com, 10.3M OTB + 7.6M online games) is CC BY-NC-SA 4.0 — NON-COMMERCIAL, cannot ship in a commercial app. Lichess Elite Database (nikonoel) and PGN Mentor state NO explicit license — ambiguous, avoid relying on them for redistribution; rebuild equivalents from the CC0 Lichess dumps.
- Lichess opening-name detection algorithm (scalachess OpeningDb, the reference impl to mirror offline): byFen: Map[openingFEN, Opening]; findByStandardFen does a direct map.get; searchInPositions walks the game's positions, converts each to opening-FEN (Fen.writeOpening = placement+turn+castling+ep), looks each up, and RETURNS THE DEEPEST/LAST match (iterates back to front), stopping when the board has <20 pieces (openings don't extend into endgames). This is exactly reproducible with the chess-openings dist EPD column as the map key.

**Recommendations**
- BUILD STEP (ship-time, offline at runtime): vendor github.com/lichess-org/chess-openings, run `pip install chess>=1,<2 && make` (or run bin/gen.py directly) to produce dist/all.tsv with columns eco,name,pgn,uci,epd. Parse all.tsv at build time into a single lookup table keyed by the EPD string. This is the entire name database — ~3,733 rows, a few hundred KB, trivially bundled. CC0 means no attribution obligation, but crediting Lichess is good practice.
- RUNTIME NAME DETECTION: maintain the position after each played move. Compute its 4-field key = piece placement + side-to-move + castling + en-passant, applying the 'en-passant only if a legal ep capture exists' rule. Look up that key in the EPD->name map. To label the whole game, walk all positions and keep the DEEPEST (last) hit, ignoring positions below ~20 pieces — mirroring scalachess. Cache the running key incrementally (Zobrist-style) rather than re-serializing the board each ply.
- Use a single position-normalization function shared by BOTH the name lookup (EPD string key) and the Polyglot book (Zobrist 64-bit key) so transpositions behave identically. Reuse a vetted library to avoid Zobrist/EPD bugs: in an Electron/Node app, `chessops` (TS, by lichess author niklasf, MIT) provides FEN/EPD and Polyglot-compatible hashing; chess.js can track positions but verify its FEN ep handling matches the 'legal-ep-only' rule before using it as a key.
- BOOK MOVES: ship a Polyglot .bin and read it with a tiny binary-search reader over 16-byte big-endian records (struct >QHHI). For a position, compute its Polyglot Zobrist key, binary-search the first entry with that key, scan forward collecting all entries sharing it, decode each move (remember the king-captures-rook castling rewrite), and present moves ranked by weight. Show weight as a relative popularity bar. This gives transposition-aware book lines for free.
- GENERATE YOUR OWN BOOK from CC0 data to dodge license ambiguity: download a few months of database.lichess.org standard rated PGNs (CC0), filter to high rating (e.g. both players 2200+/2400+) and standard time controls, then build a .bin with the `polyglot make-book` tool (or python-chess to emit entries) capping depth to ~12-16 plies. Weight = game count (optionally score-weighted). Result is a fully redistributable, commercial-safe opening book + you can derive per-move W/D/L stats for the explorer panel.
- EXPLORER STATS PANEL (chess.com/lichess-style): precompute, per opening-FEN, the aggregated next-move stats (times played, white win %, draw %, black win %) from the same CC0 Lichess PGN filter used for the book. Store as a compact on-disk table (e.g. SQLite keyed by EPD, or a sorted binary keyed by Zobrist) so a position lookup returns: opening name+ECO, candidate moves with counts and result percentages, and links to continue. Keep it memory-mapped/indexed for instant offline queries.
- DATA STORAGE: for the 3.7k-row name set, an in-memory JS Map (loaded from a bundled JSON/TSV) is plenty. For the larger move-stats/explorer table, use better-sqlite3 (synchronous, fast, bundles into Electron) with a PRIMARY KEY on the EPD/key column, or a custom sorted binary file with binary search if you want zero dependencies. Both are fully offline. Bundle the .bin book as a static asset.
- TESTING: use python-chess `chess.polyglot.zobrist_hash(board)` and `board.epd()` as golden oracles in your build/test pipeline — assert your JS/TS key generator produces identical EPD strings and identical 64-bit Polyglot keys for a battery of positions, INCLUDING the en-passant edge cases (1.e4 e5 2.Nf3 must yield ep '-'; 1.e4 Nf6 2.e5 d5 must yield ep 'd6') and all four castling-rights transitions. A mismatch here silently breaks both name detection and book hits.

**Risks**
- EN-PASSANT MISMATCH (highest risk): if your runtime FEN/EPD or Zobrist generator always emits an ep target after any pawn double-push, keys won't match the CC0 dataset or standard Polyglot books and lookups will silently miss. The rule is 'ep field set only when a legal ep capture exists.' Verified: 1.e4 e5 2.Nf3 -> ep '-'. Test this explicitly.
- CASTLING-MOVE ENCODING in Polyglot: book moves for castling are stored as king-captures-own-rook (e1h1/e1a1/e8h8/e8a8). If you apply the raw decoded move to a board expecting standard e1g1/e1c1 you'll get an illegal move or wrong square. Detect a king moving onto its own rook and rewrite to the two-square king move (and the Chess960 case if you ever support it).
- LICENSE TRAPS: Lumbra's Gigabase is CC BY-NC-SA 4.0 (NON-COMMERCIAL) — must NOT be bundled in a commercial app. Lichess Elite (nikonoel) and PGN Mentor state no explicit license — redistribution is legally ambiguous; do not ship them. Chess.com assets/data are proprietary. Mitigation: source ALL games from the explicitly-CC0 database.lichess.org and generate your own book/stats.
- ECO/name coverage is finite (~3,733 lines): deep or offbeat lines run past the named tree and will return no opening name. Expected behavior — fall back to the deepest matched ancestor name (the longest-prefix/last-match logic gives this automatically), and don't error when a position has no entry.
- Spec-mirror endianness/struct bugs: Polyglot is BIG-endian (>QHHI). Reading it with native little-endian assumptions corrupts every field. Also the 781-entry POLYGLOT_RANDOM_ARRAY constants must be byte-for-byte the standard ones (the same array python-chess/chessops ship) or your keys won't match any third-party .bin. Reuse a library's array rather than hand-typing it.
- Name 'detection by longest move-sequence prefix' vs 'by position lookup' are subtly different: lichess does POSITION (FEN) lookup, which means transpositions into a named position get the name even via a different move order — usually desired. If you instead match the literal PGN move string you'll miss transpositions. Use the EPD/position-keyed approach, not raw move-string matching.
- Build-time Python dependency: generating dist/ requires python-chess. Pin `chess>=1,<2` (current 1.11.2). If you'd rather avoid Python in CI, either commit the generated dist/all.tsv as a vendored artifact or regenerate the uci/epd columns with chessops in a Node build script — but then re-validate against the python-chess oracle.
- Polyglot weight semantics vary by book author (raw game counts vs. tuned values vs. score-weighting). Don't assume weights are comparable across different .bin files or directly equal to popularity. If you display percentages, derive them from your own generated stats table, and treat third-party book weights only as relative ordering hints.

- GAME ACCURACY (whole game): (1) sliding windows sized by game length; (2) per-window volatility = std-dev of Win%; (3) volatility-weighted mean of move accuracies; (4) harmonic mean of move accuracies; (5) final = average of the weighted mean and the harmonic mean. Dampens single blunders / one-sided endgames.
- LICHESS MOVE CLASSIFICATION (open thresholds): Inaccuracy = move ≥0.5 pawn (50cp) worse than best; Mistake = ≥1.0 pawn (100cp); Blunder = ≥3.0 pawn (300cp) worse. Suppressed when you're already clearly winning and merely pick a slightly smaller advantage (cap eval before diffing).
- CHESS.COM CLASSIFICATION (Expected-Points model, for richer badges): expected points 0.00=lost,0.50=even,1.00=won. Best=0.00 EP lost; Excellent=0.00–0.02; Good=0.02–0.05; Inaccuracy=0.05–0.10; Mistake=0.10–0.20; Blunder=0.20–1.00. - NAG GLYPHS (PGN standard): $1=! good, $2=? mistake, $3=!! brilliant, $4=?? blunder, $5=!? interesting/speculative, $6=?! dubious. ## JS/TS chess libraries (chess.js vs chessops, chessground board UI, PGN parsing, EPD hashing, eval-graph charting) for a React + TypeScript + Electron offline analysis/teaching app.

**Summary.** For a React + TypeScript + Electron chess analysis and teaching app that needs full game-tree analysis (variations, NAGs, comments), the strongest stack is chessops + chessground as the core, with chess.js as an optional lightweight fallback only.

Recommended set:
- Rules / legality / SAN / FEN / move generation: chessops (preferred) or chess.js (simpler, fallback). chessops is the lichess-grade engine-side library; chess.js has a smaller, friendlier API but a critical limitation for an analysis app (see below).
- Board rendering / interaction: chessground (the actual lichess board UI). It is rendering-only with NO chess logic and NO dependencies, so it pairs with whichever rules library you pick.
- PGN import/export with variations + NAGs + comments: chessops `pgn` module. This is the decisive reason to adopt chessops. chess.js 1.x DISCARDS recursive annotation variations (RAV) on PGN load, so any analysis/variation tree in the PGN is lost — disqualifying it as the primary parser for an analysis/teaching tool. chessops parses a full game tree (variations, NAGs, comments, starting comments) and has a DoS-resistant streaming parser (PgnParser) plus `makePgn` for export.
- Opening identification / position keying ("EPD hashing"): use chessops EPD. chessops produces EPD via `makeFen(setup, { epd: true })` (and exposes `INITIAL_EPD` / `EMPTY_EPD`). The EPD string (board + turn + castling + en passant, no move counters) is the correct stable key for transposition-aware opening lookups; index it in a Map or SQLite. chessops does NOT bundle a Zobrist/Polyglot hash module (no hash.ts / polyglot module in current source), so if you specifically need 64-bit Polyglot book hashes for a .bin opening book, add a small dedicated Polyglot library or port the Zobrist constants — otherwise EPD-string keys are sufficient and simpler.
- Eval graph / accuracy charting: lichess itself renders its eval graph with custom hand-rolled SVG (no third-party chart lib) via its TypeScript UI. For an Electron app you can either (a) hand-roll an SVG/Canvas area chart (best control over the lichess-style red/grey advantage fill, lightweight, no deps) or (b) use a React charting lib bundled offline — Recharts (MIT) for quick declarative charts, or visx/D3 (MIT/ISC) for a fully custom advantage chart. All chart libs bundle fine offline; pick based on how custom the styling must be.

React + Electron integration: wrap chessground yourself in a thin React component (init in useEffect/useRef, call `cg.set(...)` on prop changes, `destroy()` on unmount). Do NOT depend on @react-chess/chessground — it is stale (last release Sep 2022, React <=18 only, GPL-3.0) and lags far behind chessground 9.x. A ~40-line in-house wrapper is the lichess-blessed pattern and keeps you current.

Licensing note for offline bundling: chessground is MIT (clean). chess.js is BSD-2-Clause (clean). chessops is GPL-3.0-or-later — bundling it makes the Electron renderer a GPL-covered work, which is fine for an open-source app but is a real obligation to flag if the app is intended to be proprietary/closed-source. If GPL is unacceptable, fall back to chess.js (BSD) for rules+SAN and pair it with a permissively-licensed PGN-variation parser (e.g. @mliebelt/pgn-parser, MIT) to recover variation support — but that is a more bespoke path than chessops' integrated tree.

## LOCAL (no-LLM) coaching / explanation engine

**Summary.** A fully offline coaching engine is achievable by composing four deterministic layers, all of which Lichess already ships as open source (AGPL) and which port cleanly to a chess.js (BSD) + bundled-Stockfish Electron app with NO LLM. (1) CLASSIFY MAGNITUDE: convert each move's Stockfish cp/mate eval to a "winning chances" value via the exact Lichess sigmoid Win% = 50 + 50*(2/(1+exp(-0.00368208*cp)) - 1), clamp cp to [-1000,1000], map mate to (21 - min(10,|mate|))*100 cp; the POV-signed DROP in win-chances (delta, 0..1) is bucketed into Inaccuracy/Mistake/Blunder. Lichess analysis uses delta thresholds 0.10/0.20/0.30 (Advice.scala); the interactive Practice tool uses TIGHTER thresholds on the same shift: <0.025 good, <0.06 inaccuracy, <0.14 mistake, else blunder (practiceCtrl.ts). Mate transitions (cp->mate / mate->cp) get special MateCreated/MateLost handling with their own severity bands. (2) DETECT WHAT CHANGED: compute material_diff before/after, run a cheap static scan (board.attackers / board.attacks / SquareSet.between / board.pin equivalents) on the resulting position to find hanging pieces, king-safety change, new threats, mate threats. (3) DETECT MOTIFS from the engine PV: re-implement Lichess's puzzle 'cook' detectors (fork, skewer, discovered attack/check, double check, back-rank mate, trapped piece, hanging piece, deflection, interference, pin-prevents-attack/escape, capturing-defender, x-ray) which are pure geometric + short-lookahead boolean functions over the PV node list; they rely on a small set of primitives (values, king_values, ray_piece_types, is_hanging/is_defended, is_in_bad_spot, is_trapped, attacked_opponent_squares). (4) GENERATE TEXT via slot-filling templates keyed by (verdict x motif x material/king-safety facts), reusing Lichess's own AGPL motif definitions (puzzleTheme.xml) and Learn/Practice praise vocabulary. Positional commentary (space, development, pawn structure, open files, outposts) is produced from classic chessprogramming.org static-eval terms computed offline. NOTE: Lichess's own 'overloading' detector is a stub (returns False) and is approximated by 'deflection' — flag this as a known gap to implement yourself.


**Key facts**
- WIN% (Lichess canonical): Win% = 50 + 50 * (2 / (1 + exp(-0.00368208 * centipawns)) - 1). Implemented in ui/lib/src/ceval/winningChances.ts as rawWinningChances(cp)=2/(1+exp(-0.00368208*cp))-1, returning -1..1; cp is clamped to [-1000,1000] before the sigmoid.
- MATE -> win% mapping: cp = (21 - min(10, |mate|)) * 100, signed by mate's sign, then fed through rawWinningChances. So a forced mate maps to a very high (but finite, <1) win-chance band. (winningChances.ts mateWinningChances).
- ANALYSIS annotation thresholds (modules/tree/src/main/Advice.scala, CpAdvice.winningChanceJudgements): on POV-signed win-chances delta -> 0.30 = Blunder, 0.20 = Mistake, 0.10 = Inaccuracy. delta = (currentWinChances - prevWinChances) signed for the moving color; first bucket whose threshold <= delta wins.
- PRACTICE-tool thresholds are different and tighter (ui/analyse/src/practice/practiceCtrl.ts makeComment): shift = -povDiff(...); verdict = shift<0.025 'goodMove', <0.06 'inaccuracy', <0.14 'mistake', else 'blunder'. povDiff already divides the chances difference by 2, so this 0..1 scale is not identical to Advice.scala's delta. If the played move == engine best (or equivalent castling), verdict is forced to 'goodMove'.
- MATE-sequence advice (Advice.scala MateAdvice): cp->mate(negative) = MateCreated; mate(positive)->cp = MateLost; mate(pos)->mate(neg) = MateLost. Severity by prior/after cp: MateCreated severity = Inaccuracy if prevPovCp < -999, Mistake if < -700, else Blunder; MateLost = Inaccuracy if povCp > 999, Mistake if > 700, else Blunder. MateDelayed (worse mate) is NOT annotated.
- Lichess COMMENT template (Advice.scala makeComment): optionally '(<evalBefore> -> <evalAfter>) ' + (mate: 'Checkmate is now unavoidable' / 'Lost forced checkmate sequence' / 'Not the best checkmate sequence', OR cp: the Judgement word) + '.' + ' <bestMoveSan> was best.' This is the entire user-facing string -- a proven minimal template you can extend.
- ACCURACY% per move (lichess.org/page/accuracy): Accuracy% = 103.1668 * exp(-0.04354 * (winPercentBefore - winPercentAfter)) - 3.1669, clamped 0..100. Game accuracy = average of (a) volatility-weighted mean and (b) harmonic mean of per-move accuracies, using a sliding window of Win% std-dev for weights.
- CORE PRIMITIVES (lichess-puzzler tagger/util.py, AGPL): values={P:1,N:3,B:3,R:5,Q:9}; king_values adds K:99; ray_piece_types=[Q,R,B]. is_defended(): square has a same-color attacker OR (ray-defense) removing a ray attacker reveals a same-color defender. is_hanging = not is_defended. can_be_taken_by_lower_piece = an enemy non-king attacker of lower value exists. is_in_bad_spot = has enemy attacker AND (is_hanging OR can_be_taken_by_lower_piece). These four predicates power almost every motif.
- FORK detector (cook.py fork): for a player move (not by king) landing on a square that is NOT in a bad spot, count attacked enemy pieces (excluding pawns) where either king_values[target] > king_values[mover] (forking a more valuable piece) OR target is_hanging and not also defended by the moving piece's square; if count > 1 -> fork.
- SKEWER detector (cook.py skewer): after an opponent ray-piece move, the player captures on opp.to_square with a ray piece (Q/R/B), opp moved INTO the between-squares of the capture line, king_values[opp piece] > king_values[captured], and the capture lands in a bad spot -> skewer (i.e. the more valuable piece was in front and moved, exposing the lesser behind).
- DISCOVERED ATTACK / CHECK (cook.py): discovered_check = a checker exists that is NOT the square the player just moved to. discovered_attack = discovered_check OR a capture whose from->to between-squares contain the previous player move's from_square (the piece that vacated the line), with the unveiled line distinct from the moved piece. DOUBLE CHECK = board.checkers() count > 1.
- BACK-RANK MATE (cook.py back_rank_mate): final position is checkmate, mated king on its back rank, the 2-3 escape squares one rank in front are all blocked by the king's own pieces or attacked, and at least one checker sits on the back rank. TRAPPED PIECE uses is_trapped(): a non-pawn/non-king piece in a bad spot with no legal escape to a non-bad square and no equal-or-better capture available.
- HANGING PIECE (cook.py hanging_piece): the piece captured on the first player move was a non-pawn piece that was is_hanging in the prior position (and material is retained afterwards). DEFLECTION and INTERFERENCE detectors capture a piece that is_hanging only because a defending RAY piece's defense line was blocked/distracted (interfering piece landed in SquareSet.between(target, defender)).
- PIN detection: python-chess board.pin(color, square) returns the pin-direction mask (BB_ALL if not pinned); board.is_pinned(color, square) is the boolean. cook.py builds pin_prevents_attack (pinned enemy piece can't defend a higher/ hanging player piece) and pin_prevents_escape (pinned enemy piece can't flee the attacker on the pin line). For a JS port, chess.js v1.4 exposes attackers(square, color) and isAttacked(square, color) (pinned pieces still count as attackers), but NOT a direct pin() -- pins must be derived via ray scans + king alignment.
- chess.js (BSD-2-Clause) gives the offline JS primitives needed in Electron: isAttacked(square,color):boolean and attackers(square, attackedBy?):Square[] (added/released in the v1.x line, ref PR #456 / issue #464). This lets you re-implement is_hanging/is_defended/attacked_opponent_squares without python-chess or a Python runtime.
- OPEN-SOURCE template reference python-chess-annotator (GPL-3.0): uses winning_chances(cp)=50+50*(2/(1+exp(-0.004*cp))-1) (note -0.004, a coarser constant than Lichess's -0.00368208), NEEDS_ANNOTATION_THRESHOLD=7.5 win%, NAG bands via ERROR_THRESHOLD BLUNDER/MISTAKE/DUBIOUS on cp delta, eval_human() emits 'Mate in N' or '%.2f' pawn eval, and var_end_comment() emits terminal-state strings: 'Stalemate', 'Insufficient material to mate', 'Fifty move rule', 'Three-fold repetition'. - LICHESS 'LEARN' praise vocabulary (translation/source/learn.xml, AGPL text): success interjections 'Awesome!', 'Excellent!', 'Great job!', 'Perfect!', "You're good at this!", 'Nailed it.'; failure 'Retry'; completion 'Congratulations! ...' lines per skill. - LICENSE CONTAGION: lichess-puzzler (cook.py/util.py) and lila (Advice.scala) are AGPL-3.0; python-chess and python-chess-annotator are GPL-3.0. Mitigation: re-implement from the documented algorithm/formulas (facts/math are not copyrightable) in BSD/MIT modules; or accept (A)GPL for the whole app. Stockfish itself is GPL-3.0 -- bundle it as a separate UCI process, do not statically link into closed code.
- WIN% CONSTANT DRIFT: there are TWO different sigmoid constants in the wild -- Lichess current -0.00368208 (winningChances.ts) vs python-chess-annotator's older -0.004. They give different thresholds. Pick ONE (recommend Lichess's -0.00368208 to match user expectations) and use it everywhere; mixing them makes verdicts inconsistent between modules.
- THRESHOLD MISMATCH BETWEEN MODES: Advice.scala delta (0.10/0.20/0.30) and practiceCtrl.ts shift (0.025/0.06/0.14) operate on win-chances scales that differ by a factor of ~2 because povDiff divides by 2. If you reuse a helper, confirm whether it already halves the difference, or your blunder threshold will be off by 2x.
- MOTIF FALSE POSITIVES/NEGATIVES: the cook.py detectors were tuned for tactical PUZZLE positions (forcing lines), not arbitrary game positions. Run on a random middlegame they can over- or under-fire (e.g. fork() ignores pawn targets and requires the landing square not be in a bad spot). Mitigation: gate motif claims behind a material/eval-delta sanity check (only assert 'fork wins material' if the engine eval actually swings), and always fall back to the generic best-move template.
- OVERLOADING IS UNIMPLEMENTED upstream (cook.py returns False) -- if you advertise 'overloaded piece' detection you must build it yourself; relying on 'deflection' as a proxy will miss many overload cases and mislabel some.
- STATIC 'WHAT CHANGED' SCANS CAN MISLEAD without search: a piece that looks 'hanging' to a static attackers() scan may be defended tactically (pin, counter-threat, in-between move). Lichess mitigates partly via is_defended's ray-reveal trick, but the robust signal is still the engine eval delta. Treat static motif detection as EXPLANATION of an eval swing the engine already found, not as an independent oracle.
- chess.js pin support gap: chess.js exposes attackers()/isAttacked() but has no direct board.pin()/is_pinned() equivalent like python-chess; absolute pins must be derived via ray scans toward the king. Budget engineering time for a correct pin/skewer ray module, the trickiest primitive to port.
- PV DEPTH/STABILITY: motif narration of the best line depends on a stable PV. At shallow depth or in sharp positions the PV can be noisy or change between runs, producing inconsistent explanations. Fix depth/nodes and cache to keep coaching text reproducible.
- TEMPLATE REPETITIVENESS: pure templates risk sounding robotic ('was best.' on every move). Without an LLM, mitigate with multiple surface variants per (verdict,motif) cell and by only commenting when something noteworthy changed -- but accept the coaching voice will be more formulaic than chess.com's Game Review; set product expectations accordingly.
- ACCURACY-FORMULA EDGE CASES: the accuracy exp() can exceed 100 or go negative for tiny/huge win% swings; clamp to [0,100]. Game-accuracy's volatility-weighted + harmonic-mean blend is under-documented publicly -- expect to tune the sliding-window size empirically to match Lichess numbers if exact parity matters.

## Electron architecture, security & packaging

**Summary.** Use electron-vite (a CLI wrapper over Vite, not raw vite-plugin-electron) with React + TS for the three-process split: main, preload, and renderer each get their own Vite build with HMR for the renderer and hot-reload for main/preload. Lock security to Electron defaults: contextIsolation=true, nodeIntegration=false, sandbox=true, webSecurity=true; expose a minimal typed API via contextBridge + ipcRenderer.invoke/ipcMain.handle (never the raw ipcRenderer), validate every IPC sender origin and arg schema, and ship a strict CSP (via onHeadersReceived for app-protocol content, with a meta-tag fallback). Package with electron-builder (better-documented than Forge for resource exclusion) producing a Windows NSIS installer plus a single-file portable .exe. Ship Stockfish (GPL-3.0 binary, NNUE embedded), puzzles.sqlite, and any native module (better-sqlite3) OUTSIDE the asar: data/binaries go in extraResources, native .node modules in asarUnpack, resolved at runtime from process.resourcesPath. Keep ALL runtime/user data in app.getPath('userData'); in DEV redirect userData into the project via app.setPath('userData', ...) before app 'ready' (gated on !app.isPackaged) so nothing leaks to the Desktop. For git: commit source only, .gitignore build output + dev userData, and use Git LFS (tracked via .gitattributes from day one) for the few large binaries/DBs — or better, fetch them at build time so they never bloat history.


**Key facts**
- electron-vite is a CLI integrated with Vite that pre-configures the main/preload/renderer multi-entry split with sensible Electron defaults, out-of-the-box TypeScript + React/Vue/Svelte/Solid support, HMR for the renderer and hot-reload for main+preload; vite-plugin-electron is the lower-level plugin it builds on. For React+TS the official electron-vite-react template is the recommended start. (electron-vite.org, github.com/electron-vite/electron-vite-react)
- Electron security defaults to lean on: contextIsolation=true (default since 12.0), nodeIntegration=false (default since 5.0), sandbox=true (default since 20.0). Disabling contextIsolation also disables the sandbox. Keep webSecurity=true and never set allowRunningInsecureContent or experimentalFeatures. (electronjs.org/docs/latest/tutorial/security)
- From Electron 20+, preload scripts are sandboxed and have NO full Node environment — only a polyfilled require with a limited API set, and you cannot split the preload via CommonJS requires; use a bundler (electron-vite does this) to emit a single preload file. (electronjs.org/docs/latest/tutorial/sandbox)
- Secure IPC pattern: expose ONE method per channel through contextBridge using ipcRenderer.invoke paired with ipcMain.handle; never pass the whole ipcRenderer over the bridge (it returns empty and is a security footgun). In each handler validate the sender origin via event.senderFrame.url and validate args against a schema. (electronjs.org/docs/latest/tutorial/ipc, context-bridge)
- CSP: set via session.defaultSession.webRequest.onHeadersReceived returning a Content-Security-Policy responseHeader (e.g. default-src 'none'; script-src 'self'; style-src 'self'; img-src 'self' data:); a <meta http-equiv="Content-Security-Policy"> tag is the fallback for file:// content. Prefer a custom protocol over file:// for serving the renderer. (electronjs.org/docs/latest/tutorial/security)
- electron-builder vs Forge: electron-vite documents electron-builder more thoroughly for resource exclusion. Forge's output dir is hard-coded to 'out' and conflicts with electron-vite unless you set outDir to 'dist'. Recommendation: electron-builder. (electron-vite.org/guide/distribution)
- Large/native assets must NOT be packed into the asar. Use extraResources (globs copied into the resources dir, reachable at runtime via process.resourcesPath) for data files and standalone binaries (stockfish.exe, *.nnue, puzzles.sqlite); use asarUnpack for native node modules (node_modules/better-sqlite3/**, out/main/chunks/*.node). electron-vite suggests putting such assets in public/ to simplify exclusion. (electron.build/docs/contents, electron-vite.org/guide/distribution)
- Windows targets: configure win.target = ['nsis','portable']. NSIS options: oneClick (set false for an assisted installer), perMachine, allowToChangeInstallationDirectory:true, createDesktopShortcut:true, artifactName. The portable target produces a single self-contained .exe and exposes PORTABLE_EXECUTABLE_DIR/FILE/APP_FILENAME env vars at runtime. (electron.build/nsis.html, electron.build/win.html)
- app.getPath('userData') is the correct location for all app + user data (settings, analysis cache, user PGNs); to change it you must call app.setPath before the 'ready' event (and override sessionData before ready too). This keeps data out of the Desktop/project. (electronjs.org/docs/latest/api/app)
- Stockfish is GPL-3.0: distributing its binary obligates you to ship the GPL license text + a pointer to the exact source used to build it. The strongest NNUE network is embedded in the official binary (exportable via the UCI export_net command), so a default build needs no separate .nnue file. (github.com/official-stockfish/Stockfish, official-stockfish/networks)
- Git LFS best practice: set up .gitattributes with LFS tracking BEFORE committing any binary, track by extension (*.nnue, *.sqlite, stockfish*.exe), and commit .gitattributes so all clones use LFS. LFS must be installed on every dev machine and hosts cap LFS storage/bandwidth. Alternative for redistributable engine/DB assets: download them at build time so they never enter git history. (git-lfs.com, atlassian.com/git/tutorials/git-lfs)
- Since electron-vite 2.0 you may set "type":"module" in package.json even on Electron <28; main+preload are then emitted as CommonJS .cjs files while the renderer uses ESM. (electron-vite.org)

**Recommendations**
- Scaffold from the official electron-vite-react TypeScript template; keep the three-entry layout (src/main, src/preload, src/renderer) and one electron.vite.config.ts with main/preload/renderer sections. This gives the secure split plus HMR/hot-reload for free.
- Set BrowserWindow webPreferences explicitly even though they are defaults: { contextIsolation: true, nodeIntegration: false, sandbox: true, webSecurity: true, preload: <built preload path> }. Treat any deviation as a security bug in review.
- Design a single typed IPC surface: in preload, contextBridge.exposeInMainWorld('api', { analyze: (fen) => ipcRenderer.invoke('engine:analyze', fen), getPuzzle: (id) => ipcRenderer.invoke('puzzles:get', id) }). Define a shared TS type for window.api consumed by the renderer. Never expose ipcRenderer, fs, child_process, or path to the renderer.
- Harden every ipcMain.handle: validate event.senderFrame origin against an allowlist (your app:// or file origin) and validate the payload with a schema (zod). Register a will-navigate handler and setWindowOpenHandler(() => ({ action: 'deny' })) to block navigation/popups.
- Serve the renderer in production via a registered custom protocol (e.g. app://) rather than file://, and apply CSP through session.defaultSession.webRequest.onHeadersReceived: default-src 'none'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; connect-src 'self'; font-src 'self'. No remote origins at all (fully offline).
- Run Stockfish as a child process (UCI over stdio) from the MAIN process only — never spawn from the renderer. Resolve its path at runtime: app.isPackaged ? path.join(process.resourcesPath, 'engine', 'stockfish.exe') : path.join(__dirname, '../../resources/engine/stockfish.exe').
- Place stockfish.exe, the .nnue (if using a non-embedded network), and puzzles.sqlite under resources/ (or public/) and add to electron-builder: extraResources: [{ from: 'resources', to: '.' }]; add asarUnpack for native modules: ['**/node_modules/better-sqlite3/**','out/main/chunks/*.node']. Open the DB read-only from process.resourcesPath; copy any writable DB into userData on first launch.
- Configure Windows build: win.target=['nsis','portable']; nsis={oneClick:false, perMachine:false, allowToChangeInstallationDirectory:true, createDesktopShortcut:true, artifactName:'${productName}-Setup-${version}.${ext}'}; portable.artifactName:'${productName}-Portable-${version}.${ext}'. Provide build/icon.ico.
- Redirect userData in DEV so nothing leaks to the Desktop: at the very top of main, before app is ready — if (!app.isPackaged) app.setPath('userData', path.join(__dirname, '../../.devdata')); add .devdata/ to .gitignore. In production leave userData at its OS default.
- Repo layout (commit source only): /src/{main,preload,renderer}, /resources (engine+db, LFS or build-fetched), /build (icons/installer assets), electron.vite.config.ts, electron-builder.yml, package.json, tsconfig*.json, .gitattributes, .gitignore. .gitignore must include node_modules/, dist/, out/, release/, .devdata/, *.log. Track large binaries with Git LFS via .gitattributes (*.nnue, *.sqlite, resources/engine/stockfish* filter=lfs diff=lfs merge=lfs -text) set up before the first binary commit — or fetch Stockfish/puzzles at build time to keep them out of history entirely.
- Satisfy Stockfish GPL-3.0: bundle the GPL-3.0 license text and a NOTICE/THIRD-PARTY file pointing to the exact official-stockfish source/tag used; surface it in an in-app About/Licenses screen. Flag any proprietary chess.com assets and substitute open Lichess/lila equivalents (board themes, piece sets, sounds).
- If you set "type":"module" in package.json, expect main/preload as .cjs; update electron-builder 'files' globs and the BrowserWindow preload path accordingly and verify the packaged app launches before relying on it.

## Local persistence + spaced repetition

**Summary.** For the offline Electron chess app, use SQLite as the single local store. Recommended driver: better-sqlite3 (synchronous, fastest, mature) compiled for Electron's ABI via @electron/rebuild and marked external in the bundler; node:sqlite is a viable zero-native-dep fallback but is still experimental and feature-thin in 2026. Bundle the Lichess puzzle DB (6,014,381 puzzles, CC0 1.0, distributed as a Zstandard-compressed CSV) by ETL-importing it into SQLite at build time. Use FSRS-6 (via the MIT-licensed ts-fsrs, which implements FSRS v6) for review scheduling of failed puzzles/mistakes — it needs ~20-30% fewer reviews than SM-2 for equal retention and degrades gracefully on early lapses. Track the user's own puzzle strength with Glicko-2 using the exact binary win/loss model Lichess uses (each attempt = one rated 'game' player-vs-puzzle). PGN import/export via chess.js 1.4.0 (BSD-2), or cm-pgn when variations/NAGs/comments must be preserved. Fast querying over millions of puzzle rows comes from composite/covering indexes on (theme, rating), keeping the heavy puzzle table read-only, and WAL mode. A complete DDL sketch and FSRS/Glicko scheduling parameters are provided below.


**Key facts**
- DRIVER: better-sqlite3 is synchronous and the fastest/simplest SQLite lib for Node; it is a NATIVE module so it must be rebuilt against Electron's ABI (Electron uses Chromium BoringSSL, different NODE_MODULE_VERSION than stock Node — e.g. mismatch errors like 'compiled against NODE_MODULE_VERSION 115 ... requires 121'). Fix with @electron/rebuild run after npm install and again in the packaging pipeline.
- BUNDLER GOTCHA: Vite/Webpack will try to bundle the .node binary and break at runtime. Mark better-sqlite3 as EXTERNAL so Electron loads the compiled binary from node_modules at runtime (electron-vite: build.rollupOptions.external; or use the Electron externals config).
- ALTERNATIVE node:sqlite — built into Node/Electron (no native rebuild, no ABI pain, nothing to ship), but as of 2026 it is still flagged EXPERIMENTAL with a narrower API (no easy backup, fewer extensions, less battle-tested). Good fallback to eliminate native-build risk; not recommended as primary for a perf-sensitive app over millions of rows.
- ALTERNATIVE sql.js — pure-WASM SQLite, zero native deps and fully portable, but the whole DB lives in memory and writes need manual export; impractical for a multi-million-row puzzle DB. Use only for tiny/ephemeral data, not the puzzle store.
- PUZZLE DATA: Lichess puzzle database = 6,014,381 puzzles (updated 2026-06-04), licensed CC0 1.0 Universal (free for commercial/redistribution, no attribution required). Distributed as a Zstandard-compressed CSV (.csv.zst).
- Lichess CSV columns (exact): PuzzleId, FEN, Moves (UCI), Rating, RatingDeviation, Popularity (100..-100), NbPlays, Themes (space-separated tags), GameUrl, OpeningTags. FEN is the position BEFORE the opponent's setup move; the first move in Moves is the opponent move, then the solution alternates.
- RATING MODEL: Lichess puzzle ratings are Glicko-2 with BINARY outcomes — each attempt is treated as one rated game between player and puzzle (solve = player wins/puzzle loses rating; fail = reverse). This is exactly what you should replicate for a local user puzzle rating: a Glicko-2 player object vs the puzzle's known Rating/RatingDeviation as the opponent.
- SRS CHOICE: FSRS-6 over SM-2. FSRS models memory as Difficulty(1-10), Stability(days-to-90%-recall), Retrievability(current recall prob); benchmarks on 500M+ Anki reviews show ~20-30% fewer reviews than SM-2 for equal retention, and separate D/S tracking means a couple early failures don't permanently wreck a card's schedule.
- FSRS LIBRARY: ts-fsrs (open-spaced-repetition/ts-fsrs) is MIT-licensed, implements FSRS v6, requires Node >=20 (fine in Electron). Core API: createEmptyCard(), FSRS scheduler with f.repeat(card,date) to preview all four outcomes and f.next(card,date,rating) to commit. Card fields: stability, difficulty, due, state, reps, lapses, last_review, elapsed_days, scheduled_days. Rating enum: Again/Hard/Good/Easy. State enum: New/Learning/Review/Relearning.
- FSRS-6 default weights w[] (21 params): 0.212, 1.2931, 2.3065, 8.2956, 6.4133, 0.8334, 3.0194, 0.001, 1.8722, 0.1666, 0.796, 1.4835, 0.0614, 0.2629, 1.6483, 0.6014, 1.8729, 0.5425, 0.0912, 0.0658, 0.1542. Default request_retention (desired retention) = 0.9. Ship these defaults; optionally re-optimize from the user's own review log later (fsrs-rs / optimizer) — do NOT hand-edit weights.
- PGN: chess.js 1.4.0 (BSD-2-Clause) handles import (loadPgn) and export (pgn(), setHeader()) for mainline games and is also your move-legality/SAN<->UCI engine. For games with variations, NAGs, and comments preserved, use cm-pgn (shaack/cm-pgn) or @mliebelt/pgn-parser, which chess.js does not fully round-trip.
- PERF over millions of rows: a composite index on (themes-derived, rating) plus covering indexes lets SQLite answer theme+rating-window queries without touching the main row (covering index can roughly 2x throughput by skipping the rowid lookup). Themes are multi-valued, so normalize into a puzzle_theme join table (puzzle_id, theme) with index (theme, rating) for fast 'random puzzle of theme T near rating R' queries.
- PRAGMAs for the app DB: journal_mode=WAL (concurrent reads while writing), synchronous=NORMAL (safe under WAL, much faster), foreign_keys=ON, and a sensible cache_size/mmap_size. For the bulk puzzle import only, temporarily use synchronous=OFF + large transactions (~250k rows/sec achievable). Consider WITHOUT ROWID only for narrow pure-key lookup tables, not the wide puzzles table.

**Recommendations**
- Use better-sqlite3 as the primary driver. Add @electron/rebuild to a postinstall hook and to the packaging step so the native binary matches the shipped Electron ABI; mark better-sqlite3 external in the renderer/main bundler config. Keep all DB access in the main process (or a utility process) behind IPC — better-sqlite3 is synchronous and must not run on the renderer.
- At BUILD time, run an ETL: decompress the Lichess .csv.zst and bulk-INSERT into the bundled puzzles SQLite file inside one big transaction (synchronous=OFF, prepared statements). Split the space-separated Themes into a puzzle_theme join table. Ship the resulting .sqlite read-only with the app; keep all user-writable state (attempts, ratings, FSRS cards, progress, settings, saved games) in a SEPARATE writable DB in app.getPath('userData') so app updates can replace the puzzle DB without touching user data. ATTACH them at runtime.
- Adopt FSRS-6 via ts-fsrs (MIT) for review scheduling. Create one FSRS Card per reviewable item (a failed puzzle, a recurring mistake pattern, or an opening line). Map the user's self-grade or solve result to Rating.Again/Hard/Good/Easy (e.g. failed/needed-hint -> Again/Hard, solved-clean -> Good, solved-fast -> Easy). Persist stability, difficulty, due, state, reps, lapses, last_review. Query 'due today' with WHERE due <= now ORDER BY due. Keep request_retention=0.9 and the default 21 weights initially.
- Implement a local Glicko-2 user puzzle rating that mirrors Lichess: store the user's rating(1500 start), RD(350 start), volatility(0.06). On each attempt, run one Glicko-2 update treating the puzzle (its bundled Rating/RatingDeviation) as the opponent and the result as binary (1=solved before any wrong move, 0=failed). Use a vetted lib (glicko2 npm or glicko2.ts) rather than hand-rolling. This also gives you difficulty-matched puzzle selection: pick puzzles near the user's current rating.
- For puzzle selection queries, index puzzle_theme(theme, rating) and puzzles(rating). To serve 'a random unsolved puzzle of theme X within +/-100 of user rating', filter on the join table by theme and a rating BETWEEN window, exclude already-mastered puzzle_ids, and pick with ORDER BY RANDOM() LIMIT 1 over the (already small) filtered set — or precompute a rating bucket column for O(1) bucketed sampling if RANDOM() over large filtered sets is too slow.
- Use chess.js 1.4.0 for PGN import/export and SAN/UCI conversion and legality. Store games as full PGN text plus extracted headers (Event, White, Black, Result, Date, ECO) in indexed columns for fast listing/search. When importing user PGN collections, parse each game with chess.js; if you must preserve variations/comments/NAGs (annotated study games), parse with cm-pgn and store the raw PGN verbatim so nothing is lost on round-trip.
- Set PRAGMAs at every connection open: journal_mode=WAL, synchronous=NORMAL, foreign_keys=ON, busy_timeout=5000, temp_store=MEMORY, and mmap_size/cache_size tuned to RAM. Run PRAGMA optimize on close and ANALYZE after the puzzle import so the query planner has stats over the millions of rows.
- Version the schema with a user_version PRAGMA and a small migrations runner so future app updates can evolve the writable DB without data loss. Bump it whenever the DDL below changes.
